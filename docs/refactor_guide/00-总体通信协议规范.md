# 智能猫眼门锁系统 — 总体通信协议规范 v6.0

> **版本：** v6.0（简化版）
> **日期：** 2026-05-05
> **状态：** 待实施

---

## 1. 系统架构

系统由四个组件构成，通过三条通信链路连接：

```
┌──────────┐          ┌──────────┐          ┌──────────┐
│   App    │◄────────►│  Server  │◄────────►│  ESP32   │◄──────►│ STM32  │
│ (手机)   │ WebSocket│ (Python) │ WebSocket│ (C++)    │  UART  │  (C)   │
└──────────┘   JSON   └──────────┘ JSON+Bin └──────────┘ 7字节  └────────┘
              链路C                链路B                链路A
```

| 链路 | 方向 | 协议 | 数据格式 |
|------|------|------|---------|
| 链路A | ESP32 ↔ STM32 | UART 9600bps | 7字节定长帧 |
| 链路B | ESP32 ↔ Server | WebSocket（主）/ MQTT+UDP（备） | JSON文本帧 + BinaryProtocol2二进制帧 |
| 链路C | Server ↔ App | WebSocket | JSON文本帧 |

## 2. 设计原则

### 2.1 核心原则

1. **Server是透明代理**：Server不转换消息格式，只做消息路由和日志记录。App和ESP32使用相同的JSON消息格式。
2. **单级ACK**：只保留"执行完成"确认，去掉"已收到"确认。
3. **无seq_id**：不携带消息序列号，不防重放。App端通过按钮防抖+串行发送避免重复/乱序。
4. **App串行发送命令**：App在前一个命令收到ack之前，不发送下一个命令。这是单级ACK且无seq_id的前提——因为ack消息不带命令标识，App通过"下一个ack就是当前命令的响应"来匹配。按钮在命令pending期间需要disabled。
5. **无超时重传**：命令未响应时用户手动重试，协议层不做自动重传。
6. **错误码统一简单**：只有 `code: 0`（成功）和 `code: 1`（失败），附带 `msg` 描述原因。

### 2.2 保留的原始功能（来自开源 xiaozhi-esp32）

以下功能保持不变，本文档不涉及：

- WebSocket hello 握手与认证
- BinaryProtocol2 二进制流媒体（音频OPUS + 视频JPEG）
- JSON 消息类型：`hello`、`tts`、`stt`、`llm`、`mcp`、`system`（reboot/start_monitor/stop_monitor）、`alert`、`local_preview`、`custom`
- MCP JSON-RPC 2.0 设备工具调用
- OTA 固件升级
- AudioService 音频管线
- Board 抽象层

### 2.3 明确不支持的功能

以下功能在本版本中**明确不实现**：

- **MQTT+UDP 备选通道**：门锁消息仅通过WebSocket传输。MQTT+UDP通道保留原始xiaozhi-esp32的音频/视频流能力，但门锁JSON消息（lock_control、dev_control、user_mgmt、所有上报）不通过MQTT传输。
- **心跳机制**：ESP32不发送heartbeat，Server不检测心跳超时。连接状态通过WebSocket本身的TCP keep-alive和断线回调来判断。
- **协议版本协商**：hello消息不受影响，不增加protocol_version字段。本次重构是协议格式的简化，不影响hello握手流程。

## 3. 统一JSON消息格式

### 3.1 命令消息（App/Server → ESP32）

所有命令使用统一结构：

```json
{
  "type": "<消息类型>",
  "<字段>": "<值>"
}
```

**不带 `seq_id` 字段。** 消息类型决定字段结构。

### 3.2 响应消息（ESP32 → Server → App）

所有命令的响应使用统一结构：

```json
{
  "type": "ack",
  "code": 0,
  "msg": "ok"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 固定为 `"ack"` |
| code | int | `0` = 成功，`1` = 失败 |
| msg | string | 人类可读的结果描述 |

### 3.3 上报消息（ESP32 → Server → App）

上报消息（状态、事件、日志）**不需要ACK**，发出即完成。

### 3.4 错误码

| code | 含义 | 使用场景 |
|------|------|---------|
| 0 | 成功 | 命令执行成功 |
| 1 | 失败 | 任何失败情况，msg字段说明具体原因 |

不再使用 0-10 错误码体系。`msg` 字段可包含任意描述文本，如 `"设备忙"`、`"指纹库已满"`、`"STM32无响应"` 等。

## 4. 消息路由规则

### 4.1 App → ESP32（命令代理）

```
App ──JSON命令──▶ Server ──JSON命令──▶ ESP32
                     │
                     └── 记录操作日志（app_id, 命令内容, 时间戳）
```

Server收到App命令后：
1. 验证App已通过hello认证
2. 检查目标ESP32是否在线
3. 记录操作日志到数据库
4. 原封不动转发给ESP32（JSON不变）

### 4.2 ESP32 → App（上报推送）

```
ESP32 ──JSON上报──▶ Server ──JSON上报──▶ 所有关联的App
                        │
                        └── 存储到数据库（如需）
```

Server收到ESP32上报后：
1. 存储到数据库（log_report、event_report需要持久化）
2. 转发给所有关联该设备的App连接

### 4.3 谁生成什么消息

| 消息类型 | 生成者 | 消费者 |
|---------|--------|--------|
| lock_control | App | ESP32 |
| dev_control | App | ESP32 |
| user_mgmt | App | ESP32 |
| face_result | Server | ESP32 |
| system (start/stop_monitor) | App | ESP32 |
| ack | ESP32 | Server → App |
| status_report | ESP32 | Server → App |
| event_report | ESP32 | Server → App |
| log_report | ESP32 | Server → App |
| door_opened_report | ESP32 | Server → App |
| user_mgmt_result | ESP32 | Server → App |
| visit_notification | Server | App |
| visitor_intent_notification | Server | App |
| device_status | Server | App |

## 5. 与旧版协议的主要差异

| 项目 | 旧版 (v5.2/v2.5/v3.0) | 新版 (v1.0简化) |
|------|----------------------|-----------------|
| ESP32↔Server ACK | 两级确认 (esp32_ack + ack) | 单级ACK |
| seq_id | 每条命令必带 | 无 |
| 防重放 | 100条FIFO缓存 | 不做 |
| 超时重传 | 3种超时+定时清理 | 不做 |
| 错误码 | 0-10体系+STM32映射 | 0/1 + msg字符串 |
| Server→App server_ack | 有 | 无 |
| App query接口 | 9种查询 | 无（App从推送获取实时数据，历史数据用HTTP API） |
| App media_download | WebSocket分片下载 | 无（改为HTTP下载） |
| App face_management | 独立消息类型 | 合并到HTTP API |
| STM32 UART ACK | ACK_OK/ACK_ERR/PING/PONG | 全部去掉 |
| ESP32 pending_commands_ | 按UART TYPE索引的待处理队列 | 无 |
| heartbeat | 有（预留） | 无 |
| App发送模式 | 可并发 | 串行（等ack后发下一个） |
| MQTT通道门锁消息 | 未实现（仅音频视频） | 不实现（明确排除） |
