## 2. 逐项说明

### E-1: `busy_` 伪代码与并发矩阵矛盾（ESP32端）

**原问题：** 文档02 §4.4.2的伪代码对所有命令统一检查 `busy_`，但§4.4.3的并发矩阵明确允许 `dev_control` 在任何时候执行。

**修正内容：**

1. 伪代码中 `dev_control` 独立处理——不检查 `busy_`，不设置 `busy_`，发完UART即回ack。

2. 新增**单App场景假设**说明：系统假定同一时间只有一个App连接控制设备。`dev_control` 的例外在单App场景下是安全的（App的串行命令模式保证不会在等待其他命令时发送dev_control）。

3. `busy_` 状态表更新——明确标注 `dev_control` "不设busy_"。

**影响范围：** ESP32端 `application.cc` 中 `HandleSmartLockJsonMessage` 的消息分发逻辑。

**实现要点：**
```cpp
// 在 busy_ 检查之前先处理 dev_control
if (type == "dev_control") {
    HandleDevControl(root);  // 不设置busy_，发完即回ack
    return;
}
if (busy_) {
    protocol_->SendAck(1, "设备忙，请稍后重试");
    return;
}
busy_ = true;
// ... 处理 lock_control / user_mgmt
```

### E-2: `user_mgmt_result` 与 `ack` 的响应匹配（App端）

**原问题：** 文档06的单命令模式设计对所有命令统一等待 `ack` 来 resolve Promise。但 `user_mgmt` 命令（add/del/query）的有效载荷在 `user_mgmt_result` 消息中（分配的ID、查询数量、具体成功/失败描述），`ack` 只有 `{code, msg}`。

**修正内容：**

1. `sendCommand()` 增加 `responseType` 参数，默认值为 `'ack'`。

2. 命令与响应类型的对应关系：

| 命令类型 | responseType | 触发resolve的消息 |
|---------|-------------|------------------|
| `lock_control` | `'ack'` | ack |
| `dev_control` | `'ack'` | ack |
| `user_mgmt` | `'user_mgmt_result'` | user_mgmt_result |
| `system` | `'ack'` | ack |

3. `user_mgmt_result` 的处理分为两步：
   - **先emit事件**：让 `App.tsx` 拿到详细信息（`val`、`category`、`command` 等），更新对应的状态（fingerprints/nfcCards/tempPasswords列表）
   - **再resolve Promise**：让调用方知道操作已完成（成功/失败），可以恢复按钮状态

**影响范围：** App端 `DeviceService.ts` 的 `sendCommand`/`handleAck`/`handleUserMgmtResult` 方法。

**实现要点：**
```typescript
// 调用侧
// 开锁：等ack
await deviceService.sendCommand(
  { type: 'lock_control', command: 'unlock' },
  5000, 'ack'
);

// 录入指纹：等user_mgmt_result
await deviceService.sendCommand(
  { type: 'user_mgmt', category: 'finger', command: 'add', user_id: 0 },
  90000, 'user_mgmt_result'
);
```

### E-3: `user_name` 字段传递路径（Server端）

**原问题：** 文档03 §5.3说 `user_name` "不转发给ESP32"，但文档04 §3.3.4说"原封不动转发"。两处矛盾。

**修正内容：**

统一为：**Server收到 `user_mgmt` 后，从消息中删除 `user_name` 字段再转发ESP32**。`user_name` 存入Server数据库（关联 `user_id`），供后续查询使用。

**影响范围：** Server端 `UserMgmtProxyHandler`。

**实现要点：**
```python
# 转发给ESP32前
msg_json.pop("user_name", None)   # 去掉App专用字段
# 存入数据库（在转发前或转发后都可以）
await db.save_user_name(device_id, user_id, user_name)
# 原封不动转发其余字段
await esp32_conn.websocket.send(json.dumps(msg_json, ensure_ascii=False))
```

### E-4: hello 响应缺少 `token` 字段（Server端 + App端）

**原问题：** 文档05 §3说App通过hello获取token用于HTTP API认证，但文档03 §4.2的hello响应中没有 `token` 字段。

**修正内容：**

hello成功响应新增 `token` 字段（JWT格式），App保存后用于HTTP API请求的 `Authorization: Bearer <token>` 头。

```json
{
  "type": "hello",
  "status": "ok",
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "device_info": {
    "online": true,
    "mode": "normal"
  }
}
```

**影响范围：**
- Server端：`app_connection.py` 的hello处理逻辑中生成token
- App端：`DeviceService.ts` 的hello处理中保存token

**实现要点（Server端）：**
```python
from core.utils.auth import AuthToken

# hello认证成功后
token = self.auth.generate_token(device_id=device_id, app_id=app_id)
response = {
    "type": "hello",
    "status": "ok",
    "token": token,
    "device_info": {"online": True, "mode": "normal"}
}
```

**实现要点（App端）：**
```typescript
// DeviceService.ts handleTextMessage 中
if (msg.type === "hello" && msg.status === "ok") {
  this.authToken = msg.token;  // 保存token
  this.emit("status", "connected");
}

// 后续HTTP API调用
async function apiCall(endpoint: string): Promise<any> {
  const res = await fetch(`${API_BASE}${endpoint}`, {
    headers: { Authorization: `Bearer ${this.authToken}` }
  });
  return res.json();
}
```

## 3. 已确认的设计决策

以下问题经讨论后确认，不属于文档错误：

| 问题 | 决策 | 理由 |
|------|------|------|
| 多App同时控制同一设备 | 不考虑 | 当前场景为单App |
| 断线重连后不重发命令 | 确认 | 用户手动重试 |
| HTTP API失败时降级本地缓存 | 确认 | offline-first模式 |
| clear命令UART帧丢失风险 | 接受 | 简化设计的有意取舍 |
| 用户管理只回复最终结果 | 确认 | 不显示"请按手指(1/3)"等中间进度 |

## 5. 给各端开发者的提示

### ESP32开发者

- 重点看 **E-1**：`dev_control` 必须放在 `busy_` 检查**之前**处理
- 确认 `dev_control` 不设置也不检查 `busy_` 标志
- 本协议基于**单App场景**设计，不需要考虑多App并发

### Server开发者

- 重点看 **E-3** 和 **E-4**：
  - `UserMgmtProxyHandler` 转发前必须 `msg_json.pop("user_name", None)`
  - Hello认证成功后必须生成并返回 `token` 字段（JWT格式）
- `user_name` 存入数据库，关联 `user_id`

### App开发者

- 重点看 **E-2**：
  - `sendCommand` 新增 `responseType` 参数
  - `user_mgmt` 命令用 `'user_mgmt_result'`，其他命令用 `'ack'`
  - `handleUserMgmtResult` 先emit事件（更新状态列表），再resolve Promise
- 重点看 **E-4**：hello成功后保存 `token`，后续HTTP API请求带 `Authorization: Bearer <token>`
