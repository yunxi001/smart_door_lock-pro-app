# ESP32 ↔ STM32 UART 通信协议规范 v6.0

> **链路：** A（ESP32 ↔ STM32）
> **版本：** v6.0（简化版，无ACK，从 v5.2 演进）
> **日期：** 2026-05-05

---

## 1. 物理层

| 参数 | 值 |
|------|-----|
| 接口 | UART1（STM32: PA9=TX, PA10=RX） |
| 波特率 | 9600 bps |
| 数据位 | 8 |
| 校验位 | 无 |
| 停止位 | 1 |
| 流控 | 无 |
| 字节序 | 大端（Big-Endian） |

## 2. 帧格式

**固定7字节：**

```text
Byte 0   Byte 1   Byte 2   Byte 3   Byte 4   Byte 5   Byte 6
[0xAA]   [CAT]    [TYPE]   [D0]     [D1]     [D2]     [CHECKSUM]
 帧头     类别     类型     └────── 3字节数据 ──────┘    校验和
```

**校验和计算：** `CHECKSUM = (CAT + TYPE + D0 + D1 + D2) & 0xFF`

**帧同步：** 接收端通过检测帧头 `0xAA` 定位帧边界。校验和不匹配的消息直接丢弃。

**空值填充：** 未使用的数据字节填 `0xFF`。

## 3. 类别定义 (CAT)

### 3.1 重要变更：CAT 0x00 已移除

旧版协议的 CAT 0x00（系统握手：ACK_OK、ACK_ERR、PING、PONG）**全部移除**。

- STM32 **不再**回复 ACK_OK 或 ACK_ERR
- ESP32 **不再**发送 PING，STM32 **不再**回复 PONG
- ESP32通过是否收到数据帧来判断命令执行结果

### 3.2 当前类别

| CAT | 名称 | 方向 | 用途 |
|-----|------|------|------|
| 0x01 | RPT | STM32→ESP32 | 上报：事件、传感器、状态、密码、开锁日志 |
| 0x02 | CMD | ESP32→STM32 | 控制命令：锁、OLED、蜂鸣器、补光灯、查询 |
| 0x03 | USER | 双向 | 用户管理：指纹、NFC、密码 |

## 4. ESP32→STM32 命令 (CAT = 0x02)

### 4.1 命令总表

| TYPE | 名称 | D0 | D1 | D2 | 说明 |
|------|------|----|----|-----|------|
| 0x10 | CMD_LOCK | 模式 | 保持秒数 | 0xFF | 开锁/关锁控制 |
| 0x11 | CMD_OLED | 图标ID | 0xFF | 0xFF | OLED显示图标 |
| 0x12 | CMD_BEEP | 次数 | 频率ID | 0xFF | 蜂鸣器控制 |
| 0x13 | CMD_SYNC_T | 时(0-23) | 分(0-59) | 秒(0-59) | 时间同步 |
| 0x14 | CMD_LIGHT | 模式 | 0xFF | 0xFF | 补光灯控制 |
| 0x80 | Q_SENSORS | 0xFF | 0xFF | 0xFF | 查询传感器（STM32回复RPT_ENV） |
| 0x81 | Q_STATUS | 0xFF | 0xFF | 0xFF | 查询状态（STM32回复RPT_STATE） |

### 4.2 CMD_LOCK (0x10)

| D0 | 含义 |
|----|------|
| 0x01 | 开锁 |
| 0x02 | 关锁 |

| D1 | 含义 |
|----|------|
| 0x00 | 默认保持3分钟 |
| 1-255 | 开锁保持秒数 |

**帧示例：**
```
开锁，保持5秒： [AA][02][10][01][05][FF][1D]
关锁：          [AA][02][10][02][00][FF][19]
```

**期望回复：** RPT_UNLOCK (0xA1) — STM32执行开锁/关锁后上报

**ESP32超时：** 发送后等待3秒，未收到 RPT_UNLOCK 则视为失败，上报 ack {code:1, msg:"STM32无响应"}

**幂等性：** STM32侧已实现。已开锁时再发开锁命令，仍然执行（电机再次转动）并上报。

### 4.3 CMD_OLED (0x11)

| D0 | 含义 |
|----|------|
| 0x00 | 清屏/待机 |
| 0x01 | WiFi已连接 |
| 0x02 | 云端已连接 |
| 0x03 | 识别中 |
| 0x04 | 识别成功 |
| 0x05 | 识别失败 |

**帧示例：**
```
显示"WiFi已连接"： [AA][02][11][01][FF][FF][1A]
```

**期望回复：** 无。ESP32发完即结束。

### 4.4 CMD_BEEP (0x12)

| D0 | 含义 |
|----|------|
| 1-255 | 响铃次数 |

| D1 | 含义 |
|----|------|
| 0x01 | 短滴 |
| 0x02 | 长鸣 |
| 0x03 | 报警音 |

**帧示例：**
```
短滴3声： [AA][02][12][03][01][FF][1F]
```

**期望回复：** 无。ESP32发完即结束。

### 4.5 CMD_SYNC_T (0x13)

同步当前时间到STM32。

**帧示例：**
```
设置时间 14:30:00： [AA][02][13][0E][1E][00][49]
```

**期望回复：** 无。ESP32发完即结束。

### 4.6 CMD_LIGHT (0x14)

| D0 | 含义 |
|----|------|
| 0x00 | 自动模式（恢复STM32光照传感器自动控制） |
| 0x01 | 强制开灯 |
| 0x02 | 强制关灯 |

**帧示例：**
```
强制开灯： [AA][02][14][01][FF][FF][1D]
恢复自动： [AA][02][14][00][FF][FF][1C]
```

**期望回复：** 期望 STM32 上报 RPT_STATE（灯状态变化），但ESP32不强制等待。

### 4.7 Q_SENSORS (0x80) 和 Q_STATUS (0x81)

查询命令，STM32用数据帧回复。

**帧示例：**
```
查询传感器： [AA][02][80][FF][FF][FF][82]
查询状态：   [AA][02][81][FF][FF][FF][83]
```

**期望回复：**
- Q_SENSORS → RPT_ENV (0xB0)
- Q_STATUS → RPT_STATE (0xB1)

**ESP32超时：** 发送后等待2秒，未收到对应数据帧则视为失败。

## 5. STM32→ESP32 上报 (CAT = 0x01)

### 5.1 上报总表

| TYPE | 名称 | D0 | D1 | D2 | 说明 |
|------|------|----|----|-----|------|
| 0xA0 | RPT_EVENT | 事件ID | 参数 | 0xFF | 事件上报 |
| 0xA1 | RPT_UNLOCK | 开锁方式 | 用户ID/剩余时间 | 结果码 | 开锁日志 |
| 0xA2 | RPT_DOOR_OPENED | 开锁方式 | 开门来源 | 0xFF | 开门日志 |
| 0xB0 | RPT_ENV | 电量% | 光照高字节 | 光照低字节 | 环境数据 |
| 0xB1 | RPT_STATE | 锁状态 | 灯状态 | 0xFF | 状态上报 |
| 0xC0 | RPT_PWD | Pwd_H | Pwd_M | Pwd_L | 密码查询响应 |

### 5.2 RPT_EVENT (0xA0)

| D0 (事件ID) | 事件名称 | D1 (参数) |
|-------------|---------|-----------|
| 0x01 | 门铃按下 | 0x00 |
| 0x02 | PIR人体检测 | 持续时间（秒） |
| 0x03 | 撬锁报警 | 报警级别（1-3） |
| 0x04 | 门未关超时 | 超时时间（分钟） |
| 0x05 | 低电量警告 | 当前电量百分比 |
| 0x06 | 关门/上锁状态 | 0x00=门关闭, 0x01=上锁成功 |

**帧示例：**
```
门铃按下：        [AA][01][A0][01][00][FF][A2]
PIR触发持续3秒：  [AA][01][A0][02][03][FF][A6]
上锁成功：        [AA][01][A0][06][01][FF][A8]
```

### 5.3 RPT_UNLOCK (0xA1)

| D0 (开锁方式) | 含义 |
|---------------|------|
| 0x01 | 指纹 |
| 0x02 | NFC |
| 0x03 | 密码 |
| 0x04 | 远程（ESP32/App） |
| 0x05 | 机械钥匙 |
| 0x06 | 临时密码 |
| 0x07 | 人脸 |

**D1 含义取决于 D2（结果码）：**

| D2 (结果) | 含义 | 此时 D1 含义 |
|-----------|------|-------------|
| 0x00 | 成功 | 用户ID（1-40），密码/远程/临时密码时为0xFF |
| 0x01-0x05 | 失败（第N次） | 0xFF（无法识别用户） |
| 0x06 | 已锁定 | 剩余锁定时间（分钟） |

**帧示例：**
```
指纹用户3开锁成功：  [AA][01][A1][01][03][00][A6]
密码开锁成功：       [AA][01][A1][03][FF][00][A4]
第2次失败：          [AA][01][A1][01][FF][02][A4]
设备锁定，剩余10分钟：[AA][01][A1][01][0A][06][BD]
```

### 5.4 RPT_DOOR_OPENED (0xA2)

| D0 | 含义（与RPT_UNLOCK相同） |
|----|------------------------|
| 0x01-0x07 | 开锁方式 |

| D1 (开门来源) | 含义 |
|---------------|------|
| 0x00 | 室外开门（开门前5秒内PIR触发过） |
| 0x01 | 室内开门（PIR未触发） |
| 0xFF | 未知 |

**帧示例：**
```
指纹开锁后室外开门： [AA][01][A2][01][00][FF][A4]
```

### 5.5 RPT_ENV (0xB0)

| D0 | D1 | D2 |
|----|----|-----|
| 电量百分比 (0-100) | 光照值高字节 | 光照值低字节 |

光照值计算：`lux = (D1 << 8) | D2`

**帧示例：**
```
电量85%，光照300 Lux： [AA][01][B0][55][01][2C][33]
```

### 5.6 RPT_STATE (0xB1)

| D0 (锁状态) | D1 (灯状态) |
|------------|------------|
| 0x00 = 锁已关 | 0x00 = 灯灭 |
| 0x01 = 锁已开 | 0x01 = 灯亮 |

**帧示例：**
```
锁关，灯亮： [AA][01][B1][00][01][FF][B2]
锁开，灯灭： [AA][01][B1][01][00][FF][B3]
```

### 5.7 RPT_PWD (0xC0)

密码查询响应，3字节Hex编码（大端）。

密码编码：6位数字 → 整数 → 3字节大端Hex
例如密码 `123456` = 0x01E240 → D0=0x01, D1=0xE2, D2=0x40

**帧示例：**
```
密码123456： [AA][01][C0][01][E2][40][C5]
```

## 6. 用户管理 (CAT = 0x03)

### 6.1 指纹管理

**ESP32→STM32 (TYPE=0x10)：**

| D0 (子命令) | D1 | D2 | 说明 |
|-------------|----|-----|------|
| 0x01 | 期望ID（0=自动分配） | 0xFF | 开始录入指纹 |
| 0x02 | 要删除的ID | 0xFF | 删除指定指纹 |
| 0x03 | 0xFF | 0xFF | 清空全部指纹 |
| 0x04 | 0xFF | 0xFF | 查询指纹数量 |

**STM32→ESP32 (TYPE=0x11)：**

| D0 (状态) | D1 | D2 | 说明 |
|-----------|----|-----|------|
| 0x01 | 当前次数(1-3) | 0xFF | 请按手指 |
| 0x02 | 0xFF | 0xFF | 请抬起手指 |
| 0x03 | 分配的ID | 0xFF | 录入成功 |
| 0x04 | 错误原因码 | 0xFF | 录入失败 |
| 0x05 | 指纹总数 | 0xFF | 数量查询响应 |
| 0x06 | 已存在的ID | 0xFF | 指纹已存在（返回已有槽位） |
| 0x07 | 新分配的ID | 原指定ID | ID被占用，自动分配了新ID |

**帧示例（完整录入流程）：**
```
1. ESP→STM： [AA][03][10][01][00][FF][15]   开始录入，自动分配ID
2. STM→ESP： [AA][03][11][01][01][FF][16]   请按手指，第1次
   （用户按压手指）
3. STM→ESP： [AA][03][11][02][00][FF][16]   请抬起手指
   （用户抬起手指）
4. STM→ESP： [AA][03][11][01][02][FF][17]   请按手指，第2次
   （用户按压手指）
5. STM→ESP： [AA][03][11][02][00][FF][16]   请抬起手指
6. STM→ESP： [AA][03][11][01][03][FF][18]   请按手指，第3次
   （用户按压手指）
7. STM→ESP： [AA][03][11][03][05][FF][1C]   录入成功，ID=5
```

**ESP32处理：**
- 发送FP_CMD后，等待FP_RESP序列
- 收到状态0x01（请按手指）= 命令已被STM32接收（隐式ACK）
- 收到状态0x03/0x04/0x06/0x07 = 录入流程结束
- 整个过程超时30秒，超时则上报 ack {code:1, msg:"录入超时"}

### 6.2 NFC管理

**ESP32→STM32 (TYPE=0x20)：**

| D0 (子命令) | D1 | D2 | 说明 |
|-------------|----|-----|------|
| 0x01 | 期望ID（0=自动分配） | 0xFF | 开始录入卡片 |
| 0x02 | 要删除的ID | 0xFF | 删除指定卡片 |
| 0x03 | 0xFF | 0xFF | 清空全部卡片 |
| 0x04 | 0xFF | 0xFF | 查询卡片数量 |

**STM32→ESP32 (TYPE=0x21)：**

| D0 (状态) | D1 | D2 | 说明 |
|-----------|----|-----|------|
| 0x01 | 0xFF | 0xFF | 请刷卡 |
| 0x02 | 0xFF | 0xFF | 请移开卡片 |
| 0x03 | 分配的ID | 0xFF | 录入成功 |
| 0x04 | 错误原因码 | 0xFF | 录入失败 |
| 0x05 | NFC卡总数 | 0xFF | 数量查询响应 |
| 0x06 | 已存在的ID | 0xFF | 卡片UID已存在 |
| 0x07 | 新分配的ID | 原指定ID | ID被占用，自动分配了新ID |

**ESP32处理：** 与指纹录入相同逻辑。超时30秒。

### 6.3 密码管理

| TYPE | 方向 | D0 | D1 | D2 | 说明 |
|------|------|----|----|-----|------|
| 0x30 | ESP→STM | Pwd_H | Pwd_M | Pwd_L | 设置全局密码（3字节Hex编码） |
| 0x31 | ESP→STM | 0xFF | 0xFF | 0xFF | 查询密码（STM32回复RPT_PWD） |
| 0x32 | ESP→STM | Pwd_H | Pwd_M | Pwd_L | 设置临时密码（第1包：密码） |
| 0x33 | ESP→STM | Exp_H | Exp_M | Exp_L | 设置临时密码（第2包：有效期秒数） |

**帧示例：**
```
设置密码123456：  [AA][03][30][01][E2][40][16]
查询密码：        [AA][03][31][FF][FF][FF][35]
```

**ESP32处理：**
- PWD_SET (0x30)：发完即结束。如果关心结果，可再发 PWD_QUERY (0x31) 验证
- PWD_QUERY (0x31)：等待 RPT_PWD (0xC0)，超时2秒
- TEMP_PWD：两包发完即结束

## 7. ESP32端处理逻辑

### 7.1 通用规则

1. ESP32发送CAT=0x02或0x03命令后，**不等待ACK**
2. 如果命令会触发STM32回复数据帧，等待对应的数据帧：
   - CMD_LOCK → 等待 RPT_UNLOCK（3秒超时）
   - Q_SENSORS → 等待 RPT_ENV（2秒超时）
   - Q_STATUS → 等待 RPT_STATE（2秒超时）
   - FP_CMD(录入) → 等待 FP_RESP序列（30秒超时）
   - FP_CMD(查询数量) → 等待 FP_RESP(status=0x05)（2秒超时）
   - NFC_CMD(录入) → 等待 NFC_RESP序列（30秒超时）
   - NFC_CMD(查询数量) → 等待 NFC_RESP(status=0x05)（2秒超时）
   - PWD_QUERY → 等待 RPT_PWD（2秒超时）
3. 不触发回复的命令（CMD_OLED、CMD_BEEP、CMD_SYNC_T、CMD_LIGHT、FP_DELETE、FP_CLEAR、NFC_DELETE、NFC_CLEAR、PWD_SET、TEMP_PWD），**发完即结束，立即向上层回复 ack {code:0}**
4. 超时后，向上层回复 ack {code:1, msg:"STM32无响应"}

### 7.2 STM32主动上报的处理

STM32随时可能主动上报以下消息（不请自来）：

- RPT_EVENT (0xA0) — 门铃、PIR、撬锁等事件
- RPT_UNLOCK (0xA1) — 本地开锁操作（指纹/NFC/密码/钥匙）
- RPT_DOOR_OPENED (0xA2) — 用户实际开门
- RPT_ENV (0xB0) — 环境数据变化（电量、光照）
- RPT_STATE (0xB1) — 状态变化（锁、灯）

ESP32收到后，转换为对应的JSON消息发送给Server。

### 7.3 无需ACK的理由

- PCB走线极短（<10cm），信号完整性极好
- 9600bps低速，时序裕量充足
- 累加和校验可检测数据损坏
- 有回复数据的命令，回复本身就是确认
- 无回复数据的命令（beep/oled/light），效果立即可见，失败可手动重试

## 8. 帧校验示例

### 校验和计算函数（C/C++）

```cpp
uint8_t calc_checksum(uint8_t cat, uint8_t type, uint8_t d0, uint8_t d1, uint8_t d2) {
    return (cat + type + d0 + d1 + d2) & 0xFF;
}
```

### 校验和计算函数（Python）

```python
def calc_checksum(cat, type_, d0, d1, d2):
    return (cat + type_ + d0 + d1 + d2) & 0xFF
```

### 验证示例

```
开锁命令：CAT=0x02, TYPE=0x10, D0=0x01, D1=0x05, D2=0xFF
校验和 = (0x02 + 0x10 + 0x01 + 0x05 + 0xFF) & 0xFF = 0x117 & 0xFF = 0x17
完整帧：[AA][02][10][01][05][FF][17] ✓

RPT_ENV：CAT=0x01, TYPE=0xB0, D0=0x55, D1=0x01, D2=0x2C
校验和 = (0x01 + 0xB0 + 0x55 + 0x01 + 0x2C) & 0xFF = 0x133 & 0xFF = 0x33
完整帧：[AA][01][B0][55][01][2C][33] ✓
```
