# 简化协议实施指南 v6.0

> **版本：** v6.0
> **日期：** 2026-05-05
> **目标：** 指导各端开发者按照新的简化协议修改代码

---

## 1. 实施总览

### 1.1 修改量估算

| 组件 | 修改文件数 | 主要操作 | 风险等级 |
|------|-----------|---------|---------|
| ESP32 | 6个 | 删除代码为主，增加并发控制 | 中 |
| 服务器 | 10-12个 | 删除+简化+新增 HTTP API | 中 |
| App | 2-3个 | 简化逻辑+串行化 | 低 |
| STM32 | **0个** | 无需修改 | 无 |

### 1.2 实施顺序

建议按以下顺序实施，每阶段完成后验证基本功能。**各阶段之间必须做集成测试**，确认无回归再进入下一阶段：

1. **第一阶段**：ESP32 — 删除 pending_commands_、esp32_ack、seq_id 相关代码
   - **验证**：用当前服务器代码测试，确认基础语音对话和门锁功能正常（服务器代码未变，应能兼容）
2. **第二阶段**：ESP32 — 简化 HandleSmartLockJsonMessage 和 HandleLockSystemMessage
   - **验证**：用当前服务器代码测试，确认所有门锁JSON消息格式正确
3. **第三阶段**：服务器 — 删除 esp32AckHandler、heartbeatHandler、error_codes.py、seq_id_cache.py；简化 commandProxyHandler、ackHandler、app_connection
   - **验证**：用已修改的ESP32和未修改的App测试（App仍发seq_id，Server应忽略）
4. **第四阶段**：App — 简化 DeviceService.ts
   - **验证**：全链路测试
5. **第五阶段**：服务器 — 实现 HTTP API（第一阶段：status + faces + visits），删除 FaceManagementHandler、清理 textMessageType 枚举
   - **验证**：App 通过 HTTP API 查询状态和人脸管理
6. **第六阶段**：服务器 — 实现 HTTP API（第二阶段：events/logs/media/password），删除 QueryHandler、MediaDownloadHandler，完成枚举清理
   - **验证**：全链路测试

### 1.3 兼容性说明

- **原始 xiaozhi-esp32 功能不受影响**：hello、tts、stt、llm、mcp、system、alert、local_preview、custom
- **BinaryProtocol2 不受影响**：音频流、视频流、人脸识别图像
- **MQTT+UDP 备选通道不受影响**
- **STM32 固件无需修改**：UART 7字节帧格式不变，只是ESP32端不再等待ACK

```python
class AckHandler(TextMessageHandler):
    async def handle(self, conn, msg_json):
        code = msg_json.get("code", 0)
        msg = msg_json.get("msg", "")
        
        # 记录日志
        if code == 0:
            conn.logger.bind(tag=TAG).debug(f"ACK 成功")
        else:
            conn.logger.bind(tag=TAG).warning(f"ACK 失败: code={code}, msg={msg}")
        
        # 直接转发给所有关联的App（附带log_report等上报消息
        await self._forward_to_apps(conn, msg_json)
    
    async def _forward_to_apps(self, conn, msg_json):
        """转发 ACK 到所有关联的 App"""
        manager = ConnectionManager.get_instance()
        app_conns = manager.get_app_conns(conn.device_id)
        if not app_conns:
            return
        msg = json.dumps(msg_json)
        for app_conn in app_conns:
            if app_conn.websocket:
                await app_conn.websocket.send(msg)
```

#### 3.3.2 LockControlProxyHandler 特殊处理

**保留 app_id 缓存逻辑**：`_forward_to_esp32` 方法中缓存 `esp32_conn.last_remote_unlock` 的逻辑**必须保留**（当前第99-109行）。`logReportHandler` 依赖此缓存来关联"谁通过 App 开了锁"。

**修改后逻辑（伪代码）：**
```python
class LockControlProxyHandler(TextMessageHandler):
    async def handle(self, conn, msg_json):
        if not hasattr(conn, "client_type") or conn.client_type != "app":
            return
        
        command = msg_json.get("command")
        
        # 检查ESP32在线
        manager = ConnectionManager.get_instance()
        esp32_conn = manager.get_esp32_conn(conn.device_id)
        if not esp32_conn or not esp32_conn.websocket:
            await self._send_error(conn, "设备离线")
            return
        
        # 记录操作日志（开锁命令）
        if command == "unlock":
            await self._log_unlock_operation(conn, msg_json)
        
        # 缓存 app_id 到 ESP32 连接对象（保留！供 logReportHandler 使用）
        if command == "unlock":
            esp32_conn.last_remote_unlock = {
                "app_id": conn.app_id
            }
        
        # 原封不动转发给 ESP32（不添加 seq_id，不重试）
        await esp32_conn.websocket.send(json.dumps(msg_json, ensure_ascii=False))
        # 不发送 server_ack，等 ESP32 的 ack 转发给 App
    
    async def _send_error(self, conn, message: str):
        """发送错误响应 — 使用统一 ack 格式"""
        await conn.websocket.send(json.dumps({
            "type": "ack",
            "code": 1,
            "msg": message
        }))
```

### 4.2 命令串行化

由于ack不带命令标识，App必须确保同一时间只有一个命令在等待ack。除了按钮防抖外，还需要实现命令串行化：

```typescript
// 方案：命令队列
class CommandQueue {
  private queue: Command[] = [];
  private processing = false;

  async enqueue(command: Command): Promise<void> {
    return new Promise((resolve, reject) => {
      this.queue.push({ command, resolve, reject });
      if (!this.processing) {
        this.processNext();
      }
    });
  }

  private async processNext() {
    if (this.queue.length === 0) {
      this.processing = false;
      return;
    }
    this.processing = true;
    const { command, resolve, reject } = this.queue.shift()!;
    try {
      this.ws.send(JSON.stringify(command));
      // 等待ack（10秒超时）
      const ack = await this.waitForAck(10000);
      if (ack.code === 0) resolve();
      else reject(new Error(ack.msg));
    } catch (e) {
      reject(e);
    }
    this.processNext(); // 发送队列中的下一个命令
  }
}
```

**注意：** 如果不需要命令队列的复杂度，也可以简单地在按钮点击后disabled，等ack回来再恢复。但命令队列更健壮——如果用户需要在开锁后自动查状态，可以入队两个命令而不用手动等待。
